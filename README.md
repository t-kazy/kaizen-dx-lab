# kaizen-dx-lab

業務改善・DX の実験場。小さな改善ネタを試して、記録して、再利用できる形に育てるリポジトリです。

## 概要

このラボでは次のようなことを行います。

- 日々の業務で感じた「もっとこうしたい」を Issue に起こす
- 手順書・テンプレート・スクリプトなど、改善の成果物を蓄積する
- 検証ログを残し、うまくいった/いかなかったを振り返れるようにする

## できること

| カテゴリ | 内容 | 置き場 |
|---|---|---|
| 手順書 | 作業手順・セットアップガイドなど | `docs/` |
| スクリプト | 自動化・ユーティリティ | `scripts/` |
| テンプレート | Issue / PR / ドキュメントの雛形 | `templates/` |
| 検証ログ | 試したことの記録・結果メモ | `logs/` |

## ディレクトリ構成

```
.
├── README.md
├── CLAUDE.md
├── docs/          # 手順書・ガイド
├── scripts/       # スクリプト・ツール
├── templates/     # テンプレート集
└── logs/          # 検証ログ・振り返りメモ
```

## 運用ルール

### ブランチ

- `main` を常にクリーンに保つ
- 作業は `feature/<内容>` や `fix/<内容>` ブランチで行い、PR 経由でマージ

### コミットメッセージ

```
<種別>: <変更内容の要約>

例:
add: 月次レポート生成スクリプト
fix: テンプレートの日付書式を修正
docs: セットアップ手順を追記
```

### 命名規則

- ファイル名はケバブケース (`setup-guide.md`, `export-csv.sh`)
- ログは日付プレフィックス (`2025-01-15-api-test.md`)

### レビュー

- 小さい変更でも PR を作り、セルフレビューまたはチームレビューを通す
- 「なぜやったか」を PR の説明に残す

## まず最初にやること

1. 改善したいネタを Issue に起こす（タイトルだけでもOK）
2. 優先度をつけて取り組む
3. 月次で棚卸し — 使われているもの・放置されているものを整理する

## 注意事項

- 社内の機密情報・個人情報をこのリポジトリに含めないこと
- 公開範囲（パブリック/プライベート）を確認したうえで運用する
- ライセンスが必要になった場合は別途 `LICENSE` ファイルを追加する

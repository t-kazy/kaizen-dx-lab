# Dify チャットフロー構成ガイド（ナレッジ検索最適化版）

13個のナレッジMDファイルを効果的に検索できるチャットフローの推奨構成。

---

## アーキテクチャ概要

```
[開始] → [質問分類] → [ナレッジ検索] → [LLM回答生成] → [回答]
```

シンプルな4ノード構成。質問分類は任意だが、精度向上に有効。

---

## ノード別設定

### 1. 開始ノード

特別な設定は不要。ユーザー入力をそのまま次のノードに渡す。

---

### 2. ナレッジ検索ノード（最重要）

ここの設定が検索精度を決める。

#### ナレッジ設定（アップロード時）

| 設定項目 | 推奨値 | 説明 |
|---------|--------|------|
| インデックスモード | **高品質** | エコノミーは精度が低い |
| Embeddingモデル | **text-embedding-3-small** 以上 | OpenAI推奨。text-embedding-3-largeがベスト |
| チャンク設定 | **カスタム** | 自動分割は使わない |
| セパレータ | `##` | H2見出し単位で分割 |
| 最大チャンクサイズ | **1000** | トークン数 |
| チャンクオーバーラップ | **100** | 前後の文脈を保持 |

#### 検索設定（チャットフロー内）

| 設定項目 | 推奨値 | 説明 |
|---------|--------|------|
| 検索モード | **ハイブリッド検索** | ベクトル+全文検索の組み合わせ |
| Rerankモデル | **有効にする** | bge-reranker-v2-m3 等 |
| Top K | **5〜8** | 取得チャンク数 |
| Score Threshold | **0.3** | これより高いと取りこぼしが増える |
| 重み配分 | セマンティック **0.7** / キーワード **0.3** | ハイブリッド検索時 |

> **「該当コンテキストがない」問題の主な原因：**
> - Score Thresholdが高すぎる（0.5以上だと厳しい）
> - 検索モードが「ベクトル検索」のみ
> - Top Kが少なすぎる（1〜2だとヒットしにくい）
> - インデックスが「エコノミー」モード

#### ナレッジの選択

13個すべてのナレッジを検索対象に含める。Difyはクエリに対して全ナレッジを横断検索し、スコア上位のチャンクを返す。

---

### 3. LLMノード（回答生成）

#### モデル設定

| 設定項目 | 推奨値 |
|---------|--------|
| モデル | gpt-4o / claude-sonnet-4-20250514 等 |
| Temperature | 0.3（事実ベースの回答向け） |
| Max Tokens | 2048 |

#### システムプロンプト

```
あなたは社内ナレッジに基づいて質問に回答するアシスタントです。

## ルール
1. 提供されたコンテキスト情報のみに基づいて回答してください
2. コンテキストに該当する情報がない場合は「該当する情報が見つかりませんでした」と正直に伝えてください
3. 回答の根拠となるドキュメント名やセクション名を明示してください
4. 推測や外部知識による補完は行わないでください

## コンテキスト情報
{{#context#}}

## ユーザーの質問
{{#sys.query#}}
```

> **重要**: `{{#context#}}` はナレッジ検索ノードの出力変数。
> ナレッジ検索ノードの出力をLLMノードのコンテキスト入力に正しく接続すること。

---

### 4. 回答ノード

LLMノードの出力をそのまま返す。

---

## トラブルシューティング

### 「該当コンテキストがない」と表示される場合

**Step 1: ナレッジ検索ノードの出力を確認**
- チャットフローのデバッグモードで実行
- ナレッジ検索ノードの出力を確認し、チャンクが返ってきているか見る
- チャンクが0件 → ナレッジ設定の問題（Step 2へ）
- チャンクがある → LLMノードの問題（Step 3へ）

**Step 2: ナレッジ設定を確認**
1. ナレッジ管理画面でドキュメントのステータスが「有効」になっているか
2. インデックスの構築が完了しているか（処理中だと検索できない）
3. チャンクが正しく分割されているか（ナレッジ > ドキュメント > チャンク一覧で確認）
4. Embeddingモデルが正しく設定されているか

**Step 3: LLMノードの接続を確認**
1. ナレッジ検索ノードの出力がLLMノードのコンテキスト入力に接続されているか
2. システムプロンプトで `{{#context#}}` を正しく参照しているか

**Step 4: 検索パラメータを調整**
1. Score Thresholdを **0.2** まで下げてテスト
2. Top Kを **10** に増やしてテスト
3. 検索モードを「ハイブリッド検索」に変更

### 検索はヒットするが回答が不正確な場合

- MDファイルのチャンク分割を見直す（見出し構造の改善）
- 各セクションにキーワードを追加する
- Q&A形式のセクションを追加する

## パフォーマンス最適化

### Rerankモデルの活用

Rerankモデルを有効にすると、検索結果の関連性が大幅に向上する。

1. Top Kを大きめ（10〜15）に設定して多めに取得
2. Rerankモデルで上位5件に絞り込む
3. LLMに渡すコンテキストが高品質になる

### マルチナレッジの検索戦略

13個のナレッジを全て1つの検索ノードで検索するのが基本。
もし特定のカテゴリの精度を上げたい場合：

```
[質問分類ノード] → 分岐
  ├─ 人事系の質問 → [HR用ナレッジ検索] → [LLM]
  ├─ IT系の質問  → [IT用ナレッジ検索] → [LLM]
  └─ その他      → [全ナレッジ検索]   → [LLM]
```

この構成は複雑になるため、まずは全ナレッジ一括検索で試し、精度に問題がある場合のみ導入する。
